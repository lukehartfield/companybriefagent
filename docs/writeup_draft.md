# Multi-Agent Competitive Intelligence System

## 1. Design Decisions

I implemented the system in `LangGraph` because the assignment is fundamentally about orchestration, routing, retries, and controlled termination rather than just prompt-writing. The chosen topology is `Supervisor-Worker`, which is also the most natural fit for the problem. A `CompetitiveIntelligenceSupervisor` receives the company name, initializes shared graph state, routes work across specialized agents, and decides whether the workflow should finalize or send the draft through one validator-driven revision loop.

The system is decomposed into five worker roles: `CompanyProfileAgent`, `FinancialAnalystAgent`, `NewsAgent`, `SynthesisAgent`, and `ValidatorAgent`. The profile agent gathers company overview and products/services information. The financial agent is intentionally split between a deterministic finance path and a best-effort competitor path: ticker resolution and `Financial Snapshot` generation are based on structured `yfinance` fields, while competitor identification can degrade independently if search is weak. The news agent gathers 2-3 recent, material developments. The synthesis agent converts the structured findings into the final six-section brief, and the validator checks for completeness, structural compliance, and likely unsupported claims. This design reflects the course theme of putting deterministic structure around probabilistic model calls.

## 2. Failure Handling

Retry and fallback are implemented in code, not just described. The supervisor wraps the profile, financial, and news agents in `_run_with_retry(...)`, which retries up to three times by default and records explicit logs such as `CompanyProfileAgent failed. Retrying (attempt 2 of 3)...` and `FinancialAnalystAgent failed after 3 attempts. Falling back to alternative strategy.` The synthesis and validator stages are handled a little differently: because they are LLM-only stages and can be expensive under free-tier rate limits, they attempt once and then drop to deterministic fallback behavior if necessary.

Fallback behavior is role-specific. The profile agent returns a partial overview instead of inventing facts. The financial agent explicitly marks structured finance fields unavailable rather than fabricating numbers, while allowing competitor discovery to degrade separately. The news agent returns a partial section with a coverage disclaimer if it cannot confidently retrieve 2-3 recent items. The synthesis agent always has a deterministic formatter fallback, so the system can still terminate cleanly even if the LLM provider is unavailable. This is the core reliability story of the project: the graph never depends on perfect tool or model availability to produce some output.

## 3. Results

The strongest intended usage is a live run on a public company with sufficient public information, because the system depends on current web, finance, and LLM endpoints. The offline fixture path uses NVIDIA so the system can still be demonstrated without live calls. In the best-case live scenario, the pipeline should produce a coherent brief covering all six required sections, with the finance section being the most deterministic and the competitor/news sections being more search-sensitive. The weakest areas are therefore likely to be competitor quality and news freshness rather than raw financial metrics.

## 4. Evaluation Reflection

If I were evaluating the system for real business use, I would separate deterministic checks from judgment-based evaluation. Deterministic checks would verify that all six sections exist, the competitor list contains exactly three distinct names, recent news contains 2-3 items with URLs, and unavailable financial data is explicitly marked unavailable rather than inferred. I would also evaluate the live system on a held-out set of public companies with different characteristics, such as a semiconductor company, a cloud software company, and a consumer company, to test whether the competitor defaults and search behavior generalize.

The main failure taxonomy would include stale news, weak competitor identification, unsupported strategic claims, malformed section structure, and finance-source failures. For subjective evaluation, I would use either a human rubric or LLM-as-judge rubric that asks whether the final brief is useful to a strategy analyst, whether the uncertainty is communicated honestly, and whether the strategic assessment is grounded in the upstream evidence. I would only consider the system good enough if it passes deterministic structure checks nearly always and produces clearly useful output across a held-out company set.
